## 1. 概述

### 1.1 本手册的目的

本手册旨在为项目团队提供一套统一的需求文档编写规范，确保项目需求文档具备以下特性：

- **一致性** — 团队内所有文档遵循相同结构和表达方式
- **可追溯性** — 每条需求可追溯到业务目标、设计实现和测试用例
- **可验证性** — 每条需求的实现与否可以被明确判定
- **完整性** — 覆盖功能、非功能、数据、接口、安全等各个方面

### 1.2 适用场景

| 场景 | 文档类型 | 详细程度 |
|------|----------|----------|
| 全新项目（中大型） | 完整版需求规格说明书 | 高 |
| 小型功能迭代 | 轻量版需求说明 | 中 |
| 紧急缺陷修复 | 需求变更单 | 低 |
| 外包开发项目 | 完整版需求规格说明书 | 高（含验收标准） |

### 1.3 核心原则

> **需求文档说的是"做什么"（What），不是"怎么做"（How）。**

- 需求文档：用户登录失败3次后锁定账户30分钟
- 设计文档：使用Redis记录失败次数，设置TTL为30分钟
- 实现文档：调用 `lockAccount(userId, 30, TimeUnit.MINUTES)`

## 2. 文档结构规范

### 2.1 标准章节划分

一份完整的企业级需求规格说明书应包含以下13个章节。项目团队可根据项目规模适当裁剪，但标注为**"必须"**的章节不可删除。

| 章节 | 名称 | 重要性 | 说明 |
|:----:|------|:------:|------|
| — | 文档控制 | 必须 | 版本记录和审批签章 |
| — | 目录 | 必须 | 自动生成 |
| 1 | 引言 | 必须 | 目的、范围、术语、参考、假设 |
| 2 | 项目概述 | 必须 | 背景、目标、范围、干系人 |
| 3 | 业务需求 | 必须 | 业务流程、业务规则 |
| 4 | 功能需求 | 必须 | 功能清单和用例描述 |
| 5 | 非功能需求 | 必须 | 性能、安全、可用性等 |
| 6 | 数据需求 | 推荐 | ER关系、数据字典、约束 |
| 7 | 接口需求 | 推荐 | 外部系统集成，非API定义 |
| 8 | 界面需求 | 推荐 | 布局、配色、交互要求 |
| 9 | 约束条件 | 必须 | 技术约束、业务约束、法规约束 |
| 10 | 风险评估 | 推荐 | 技术风险、业务风险、缓解策略 |
| 11 | 验收标准 | 必须 | 功能验收、性能验收、安全验收 |
| 12 | 需求追溯矩阵 | 推荐 | 需求与目标/接口/测试的映射 |
| 13 | 附录 | 可选 | 术语表、参考文档、变更记录 |

### 2.2 文档控制页格式

```markdown
| 版本号 | 日期 | 编写人 | 审核人 | 批准人 | 状态 | 修订说明 |
|:------:|:----:|:----:|:-----:|:-----:|:----:|---------|
| v0.1 | 2026-xx-xx | xxx | — | — | 初稿 | 初始起草 |
| v1.0 | 2026-xx-xx | xxx | xxx | — | 待评审 | 提交评审 |
| v2.0 | 2026-xx-xx | xxx | xxx | xxx | 已批准 | 评审修订 |

| 角色 | 姓名 | 签字 | 日期 |
|:-----|:-----|:----:|:----:|
| 编写 | | | |
| 审核 | | | |
| 批准 | | | |
```

**版本号规则：**

- `v0.x` — 草稿阶段，内部流转
- `v1.0` — 首次提交评审
- `v2.0+` — 评审后修订版
- `vx.1/vx.2` — 小修订（错字修正、描述澄清）

## 3. 写作规范

### 3.1 语言规范

#### 3.1.1 强制性语气词

所有需求条目必须使用以下三种语气词之一来标识约束力级别：

| 语气词 | 英文对应 | 含义 | 示例 |
|:------:|:---------|:-----|------|
| 应 | shall | 强制性需求，必须实现，验收时逐条核验 | 系统应在连续3次登录失败后锁定账户30分钟 |
| 宜 | should | 推荐性需求，建议实现，有合理理由可豁免 | 系统宜在用户首次登录时引导修改初始密码 |
| 可 | may | 可选性需求，有条件时实现 | 系统可支持指纹识别登录方式 |

**禁止使用以下模糊表述：**

- 应该、需要、最好、尽可能、大概、也许
- 系统将...（"将"表达的是动作而非约束）
- 我们想要...（人称问题）

#### 3.1.2 句法要求

每条需求应满足：

1. **单一性** — 一条需求只描述一个功能点

   > 错误：系统应支持用户登录和角色管理
   > 正确：
   > - 系统应支持用户通过账号密码登录
   > - 系统应支持管理员对角色进行增删改查

2. **陈述句** — 使用"主语 + 应/宜/可 + 谓语 + 宾语"结构

   > 正确：系统应在用户提交请假申请后自动计算请假天数

3. **无歧义** — 任何人读完需求后理解应完全一致

   > 歧义：系统应快速响应
   > 明确：系统应在正常网络条件下于2秒内完成页面加载

4. **可验证** — 存在客观方法判断需求是否被满足

   > 不可验证：系统应易于使用
   > 可验证：新用户应在3次点击内完成请假提交操作

### 3.2 格式规范

#### 3.2.1 禁止使用的元素

| 禁止 | 原因 | 替代方案 |
|------|------|----------|
| 装饰性Emoji | 降低专业性，PDF/打印可能乱码 | 使用编号和层级 |
| HTML标签 | 降低可移植性 | 使用纯Markdown语法 |
| 截图代替文字 | 无法检索和diff | 文字描述为主，截图放在附录 |
| "等等"、"之类" | 范围不明确 | 明确列出或使用"包括但不限于" |

#### 3.2.2 表格对齐

所有表格应使用统一的列对齐格式：

```markdown
| 左对齐 | 居中 | 右对齐 |
|:-------|:----:|-------:|
| 内容   | 内容 | 内容   |
```

#### 3.2.3 代码块与流程图

- API示例使用 `json` 代码块
- 配置示例使用 `yaml` 代码块
- 流程图推荐使用 PlantUML 或 Mermaid（嵌入代码，可版本管理）

### 3.3 命名规范

| 对象 | 命名规则 | 示例 |
|------|----------|------|
| 模块名 | 中文简称或英文缩写 | 系统管理、OA |
| 功能名 | 动词+名词，4-8字 | 用户登录、请假申请 |
| 字段名 | 英文小写+下划线 | emp_no、real_name |
| 枚举值 | 英文小写+下划线 | pending、in_progress |
| 文件名 | PascalCase或kebab-case | UserList.tsx、api-service.ts |

## 4. 需求编号体系

### 4.1 统一编号格式

建立覆盖全文的统一编号体系，每条需求有唯一ID：

```
{类别前缀}-{模块或分类}-{序号}
```

| 类别 | 前缀 | 格式 | 示例 |
|------|:----:|------|------|
| 功能需求 | F | F-{模块}-{序号} | F-SYS-001 |
| 非功能需求 | NF | NF-{分类}-{序号} | NF-PERF-001 |
| 业务规则 | BR | BR-{序号} | BR-001 |
| 接口需求 | IF | IF-{类型}-{序号} | IF-SRV-001 |
| 数据约束 | DT | DT-{分类}-{序号} | DT-CST-001 |
| 术语定义 | GL | GL-{序号} | GL-001 |
| 假设条件 | AS | AS-{序号} | AS-001 |
| 外部依赖 | DP | DP-{序号} | DP-001 |
| 项目目标 | OBJ | OBJ-{序号} | OBJ-001 |
| 风险项 | RSK | RSK-{分类}-{序号} | RSK-T-001 |
| 参考资料 | REF | REF-{序号} | REF-001 |

### 4.2 非功能需求分类代码

| 分类 | 代码 | 说明 |
|------|:----:|------|
| 性能 | PERF | Performance |
| 安全 | SEC | Security |
| 可用性 | AVL | Availability |
| 兼容性 | CMP | Compatibility |
| 可维护性 | MNT | Maintainability |
| 可扩展性 | EXT | Extensibility |

### 4.3 编号使用规则

1. 每个ID在文档范围内唯一
2. 删除的需求ID不应复用
3. 新增需求按序递增，不插入到现有编号之间
4. 引用其他需求时使用 `[F-SYS-001]` 格式链接

## 5. 需求的分类与层级

### 5.1 三层需求模型

```
    业务需求 (Business Requirements)
    ┌─────────────────────────────────┐
    │  为什么做？ — 项目目标和业务价值   │
    └───────────────┬─────────────────┘
                    │ 分解为
                    ▼
    用户需求 (User Requirements)
    ┌─────────────────────────────────┐
    │  做什么？   — 用户视角的用例描述   │
    └───────────────┬─────────────────┘
                    │ 分解为
                    ▼
    系统需求 (System Requirements)
    ┌─────────────────────────────────┐
    │  怎么做？   — 系统的功能和非功能规格│
    └─────────────────────────────────┘
```

### 5.2 各层产物对照

| 层级 | 产物 | 编写人 | 读者 |
|------|------|--------|------|
| 业务需求 | 项目章程、商业论证 | 产品经理 | 管理层 |
| 用户需求 | 用户故事、用例图 | 产品经理 | 架构师、开发 |
| 系统需求 | 需求规格说明书（本文档） | 需求分析师 | 开发、测试 |

## 6. 优质需求的五要素

一条好的需求描述应同时满足以下五个标准（SMART原则）：

### 6.1 具体（Specific）

需求描述精确，不同读者理解一致。

> 差：系统应支持多种登录方式
> 好：系统应支持账号密码登录和企业微信SSO登录两种认证方式

### 6.2 可衡量（Measurable）

存在客观标准来判断需求是否被满足。

> 差：系统响应要快
> 好：系统应在100并发用户下API响应时间不超过500毫秒

### 6.3 可实现（Achievable）

在当前技术栈、预算和时间约束下可实现。

> 差：系统应使用AI自动审批所有申请
> 好：系统应在审批页面展示历史相似审批的通过率，供审批人参考

### 6.4 相关性（Relevant）

需求与项目目标直接相关，不超范围。

> 无关（对一个OA系统）：系统应支持视频会议功能
> 相关：系统应集成第三方视频会议服务，支持在会议预约时生成会议链接

### 6.5 可测试（Testable）

存在明确的测试方法来验证需求实现。

> 不可测试：系统应具有良好用户体验
> 可测试：新用户在首次登录后的引导流程不超过4步，每步包含不超过50字的说明

## 7. 常见错误与修正示例

### 7.1 层级混淆：需求包含了实现细节

**错误写法：**
> 接口 `POST /api/v1/auth/login` 接收JSON参数 `{"username":"admin","password":"123456"}`，
> 使用BCrypt比对密码，返回JWT令牌存储在Redis中，Key格式为 `token:userId`。

**问题：** 这段描述混入了API设计、加密算法选型、缓存策略三个层次的实现细节。

**正确写法：**
> 系统应提供用户认证服务，接收用户凭证并返回有时效性的认证令牌。令牌有效期应为24小时，过期后用户应重新认证。密码应不可逆加密存储。

### 7.2 信息缺失：只写了功能名没有细节

**错误写法：**
> F-SYS-006 | 用户管理 | P0 | 管理员对用户账号的CRUD

**问题：** 没有前置条件、业务规则、约束条件，开发人员无法判断完成标准。

**正确写法：**
> F-SYS-006 用户管理
>
> **功能描述：** 管理员对系统用户账号进行增删改查操作。
>
> **参与者：** 系统管理员
>
> **前置条件：** 管理员已登录且拥有用户管理权限
>
> **基本事件流：**
> 1. 管理员进入用户管理页面，系统展示分页用户列表
> 2. 管理员可按关键词（姓名/工号/手机号）、部门、状态筛选
> 3. 管理员新增用户时，系统校验必填字段，自动分配默认密码
> 4. 管理员删除用户时，系统执行逻辑删除，被删除用户立即失去登录权限
>
> **业务规则：**
> - 工号和用户名在系统内应保持唯一
> - 删除应为逻辑删除，数据不可物理删除

### 7.3 表意模糊：使用了模糊形容词

**错误写法：**
> 系统应快速加载页面
> 系统应支持很多用户同时在线
> 系统应保证数据安全

**正确写法：**
> 系统应在正常网络条件下于2秒内完成页面首次加载
> 系统应在1000并发用户下保持正常运行
> 用户密码应使用BCrypt算法加密存储，传输层应使用HTTPS/TLS

### 7.4 范围越界：写了不包含的功能

**错误写法：**
> 不包含范围：
> - 财务会计核算系统
> - CRM客户管理系统

（然后正文中出现了财务报销的完整详细用例）

**修正：** 确保"包含范围"和正文内容一致。报销申请涉及的是**审批流程**而非**财务核算**，只需说明"报销申请提交后进入审批流程，审批结果通知财务系统"。

### 7.5 编号混乱：多种编号体系并存

**错误写法：**
> 规则编号 BR-001，功能编号 SYS-001，性能编号直接用"1"，安全编号用"S1"

**正确写法：** 全文统一使用4.1节规定的编号前缀体系。

## 8. 文档审查清单

文档完成后，按以下清单逐项自查：

### 8.1 结构完整性

- [ ] 包含文档控制页（版本记录 + 审批签章）
- [ ] 包含引言（目的、范围、术语、参考、假设）
- [ ] 包含项目概述（背景、目标、范围、干系人）
- [ ] 包含业务需求（流程、规则）
- [ ] 包含功能需求（清单 + 全部P0用例详情）
- [ ] 包含非功能需求（性能、安全、可用性、兼容性）
- [ ] 包含验收标准（功能、性能、安全、文档）
- [ ] 包含需求追溯矩阵
- [ ] 包含附录（术语表、参考文档、变更记录）

### 8.2 内容规范性

- [ ] 全文无装饰性Emoji
- [ ] 全文无HTML标签
- [ ] 每条需求使用"应/宜/可"注明约束力
- [ ] 每条需求具有唯一ID
- [ ] 表述无歧义（两个读者理解一致）
- [ ] 无实现细节混入（不说API路径、不说具体算法）
- [ ] 所有专业术语在术语表中定义
- [ ] 所有缩写首次出现时给出全称

### 8.3 需求质量

- [ ] 每个P0功能有完整的用例描述（参与者、前后置条件、事件流、规则、异常）
- [ ] 每条需求可验证（存在客观的通过/不通过判定标准）
- [ ] 每条需求单一（一条需求只描述一件事）
- [ ] 无重复或冲突的需求
- [ ] 优先级划分合理（P0:上线必须 / P1:上线后首个迭代）

### 8.4 追溯性

- [ ] 追溯矩阵中每条功能需求关联到业务目标
- [ ] 追溯矩阵中每条功能需求关联到对应接口
- [ ] 追溯矩阵中每条功能需求预留测试用例ID位置
- [ ] 业务规则在功能需求中有对应的引用

### 8.5 边界清晰

- [ ] 明确列出包含范围和不包含范围
- [ ] 接口需求仅描述"需要什么服务能力"，不包含具体API定义
- [ ] 数据需求仅描述实体关系和约束，不包含索引设计
- [ ] 界面需求仅描述布局和交互要求，不包含具体设计稿

### 10.3 优先级定义

| 优先级 | 含义 | 上线要求 |
|:------:|------|----------|
| P0 | 核心功能，系统上线前必须完成 | 100%实现 |
| P1 | 重要功能，上线后首个迭代完成 | 上线后2周内实现 |
| P2 | 增强功能，后续迭代逐步完善 | 有资源时实现 |
| P3 | 锦上添花，远期规划 | 待排期 |

### 10.4 风险影响程度定义

| 级别 | 含义 | 示例 |
|:----:|------|------|
| 极高 | 可能导致项目失败或重大合规问题 | 用户数据泄露 |
| 高 | 可能导致核心功能无法交付或严重延期 | 第三方接口不可用 |
| 中 | 影响部分功能或轻度延期 | 特定浏览器兼容问题 |
| 低 | 影响可控，有替代方案 | 非核心模块UI未达预期 |
